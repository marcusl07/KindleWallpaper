## Phase 10 — Stabilization, Reliability, and Performance

**T42 — Introduce explicit rotation outcomes**
Refactor wallpaper rotation orchestration into a boundary that returns explicit outcomes (e.g., success, no-active-pool, wallpaper-apply-failure) instead of relying on implicit side effects.

**T51 — Add rotation outcome/error propagation tests**
Add tests for success and failure outcomes (including no active pool and wallpaper apply failure) and verify UI-visible status propagation.

**T55 — Fix daily scheduled rotation drop when rotation in progress**
When the daily scheduler ticks and requestWallpaperRotation() returns false (busy), the scheduled rotation is silently dropped and won't retry that day. Fix by adding retry or queue behavior so the scheduled rotation eventually executes if initially rejected.

## Phase 11 — Wallpaper Collection and UX Safeguards

**T56 — Multiple background images**
Replace single background image with a managed collection. User adds/removes images, KindleWall picks randomly each rotation. Must have at least one image. UI shows collection with preview, similar pattern to Books window. Update rotation pipeline to randomly select from collection. Account for T54 cache invalidation.

**T57 — Capitalize highlight text toggle**
Add a toggle in Settings to automatically capitalize the first letter of highlight text if it is lowercase. Off by default.

**T58 — macOS wallpaper cycling conflict warning**
Add a small notice in Settings warning users to disable macOS's built-in wallpaper rotation to avoid conflicts with KindleWall managing the wallpaper.

## Open Agent Handoff Bugs

### H2 — Settings window closes on app deactivation
Reported by user after opening settings from menu bar.

Current observed behavior:
- Clicking away causes the settings window to disappear.
- User must reopen settings from the menu bar status item each time.

Expected behavior:
- Settings window remains open after losing focus.
- Window should close only when explicitly closed by user.

Repro steps:
1. Open settings from menu bar.
2. Click another app/window.
3. Observe settings window disappears instead of remaining as a normal background window.

Candidate areas to inspect:
- `App/KindleWallApp.swift` custom settings window lifecycle in `AppDelegate`.
- Window configuration flags/levels/collection behavior.
- Any implicit close/hide behavior tied to app activation policy (`.accessory`).

Acceptance criteria:
- Settings window remains visible (non-key) after app deactivation.
- User can bring it back to front without recreating state.

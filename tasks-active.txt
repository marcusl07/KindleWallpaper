## Phase 11 — Wallpaper Collection and UX Safeguards

**T56 — Background image collection data model**
Define storage model for multiple background images. Replace single image path in ScheduleSettings or AppState with a collection. Handle migration from single image to collection (existing image becomes first item).

**T57 — Background image collection UI**
Replace "Change Image" picker in Settings with a managed backgrounds view (similar pattern to Books window). Show collection with previews, add/remove controls, block removal of last image, prompt to add if empty. UI should try to follow the default macOS Wallpaper settings menu patterns.

**T58 — Random background selection in rotation pipeline**
Update WallpaperGenerator rotation path to randomly pick from the background image collection each rotation. Single image in collection = current behavior. Update T54 cache to invalidate per selected image.

**T59 — Capitalize highlight text toggle**
Add a toggle in Settings to automatically capitalize the first letter of highlight text if it is lowercase. Off by default.

**T60 — macOS wallpaper cycling conflict warning**
Add a small notice in Settings warning users to disable macOS's built-in wallpaper rotation to avoid conflicts with KindleWall managing the wallpaper.

## Open Agent Handoff Bugs

### H2 — Settings window closes on app deactivation
Reported by user after opening settings from menu bar.

Current observed behavior:
- Clicking away causes the settings window to disappear.
- User must reopen settings from the menu bar status item each time.

Expected behavior:
- Settings window remains open after losing focus.
- Window should close only when explicitly closed by user.

Repro steps:
1. Open settings from menu bar.
2. Click another app/window.
3. Observe settings window disappears instead of remaining as a normal background window.

Candidate areas to inspect:
- `App/KindleWallApp.swift` custom settings window lifecycle in `AppDelegate`.
- Window configuration flags/levels/collection behavior.
- Any implicit close/hide behavior tied to app activation policy (`.accessory`).

Acceptance criteria:
- Settings window remains visible (non-key) after app deactivation.
- User can bring it back to front without recreating state.
